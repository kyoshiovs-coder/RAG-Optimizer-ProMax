# 上下文压缩：Token不够怎么办？——从20页征信报告到核心指标

## 一、背景：当"上下文窗口"遇上"信息爆炸"

### 1.1 一个真实的信贷审批困境

某银行信贷经理使用AI辅助审批企业贷款：

**输入材料**：
- 企业征信报告：23页，15,000字
- 近3年财务报表：12页，8,000字  
- 担保物评估报告：8页，5,000字
- 行业分析报告：5页，3,000字

**模型限制**：GPT-4上下文窗口8K，实际可用约6K（扣除Prompt和输出）

**原始做法**：直接截取前6K字符（约前10页征信报告）

**后果**：
- 丢失了第15页的关键逾期记录
- 未看到第20页的担保圈风险提示
- AI给出"建议批准"，实际应"审慎拒绝"

**损失**：单笔不良贷款500万，追溯发现关键信息在截断部分。

### 1.2 Token焦虑的三种场景

| 场景 | 材料规模 | 核心矛盾 | 压缩目标 |
|:---|:---|:---|:---|
| **信贷审批** | 征信+财报+合同，3-5万字 | 全面审查 vs Token限制 | 风险信号不遗漏 |
| **电子存证** | 通话记录+聊天截图，时序长 | 完整链条 vs 窗口限制 | 关键节点保留 |
| **合规审查** | 法规+内部制度+案例，10万字+ | 精准引用 vs 检索噪声 | 引用源可追溯 |
| **投研分析** | 研报+公告+新闻，实时海量 | 时效性 vs 处理延迟 | 增量摘要更新 |

**核心洞察**：压缩不是"删减"，是"提炼"——保留决策价值密度最高的信息。

## 二、压缩策略体系：三层漏斗模型

```
┌─────────────────────────────────────────┐
│  第一层：文档级选择（Document Selection） │
│  → 哪些文档值得进入上下文？               │
│  → 方法：相关性过滤、时效性筛选、权威性排序 │
├─────────────────────────────────────────┤
│  第二层：段落级摘要（Paragraph Summarization）│
│  → 长段落如何压缩？                       │
│  → 方法：提取式摘要、生成式摘要、结构化提取  │
├─────────────────────────────────────────┤
│  第三层：句子级精选（Sentence Selection）  │
│  → 关键句如何识别？                       │
│  → 方法：重要性打分、冗余去除、逻辑链重建    │
└─────────────────────────────────────────┘
         ↓
    压缩后上下文（Token效率最大化）
```

## 三、核心技术方法

### 3.1 关键句提取：从"段落"到"金句"

**核心思想**：不是每句话都同等重要，识别"信息增益"最大的句子。

**信息重要性维度**：

| 维度 | 识别特征 | 信贷场景示例 |
|:---|:---|:---|
| **实体密度** | 包含关键实体（人、机构、金额、日期） | "2024年3月，XX企业为YY公司担保5000万" |
| **数值异常** | 与基准对比的显著偏离 | "资产负债率从45%骤升至78%" |
| **否定/警示词** | "逾期"、"违约"、"冻结"、"诉讼" | "存在2笔90天以上逾期记录" |
| **因果连接** | "因此"、"导致"、"由于" | "因行业下行，营收同比下降40%" |
| **时间敏感** | 近期事件、截止日期 | "2024年6月，贷款即将到期" |

**提取流程**：

```
原始段落（500字）
    ↓
├─ 句子切分 → [S1, S2, S3, S4, S5]
├─ 特征计算 → 实体密度、数值异常、警示词、位置权重
├─ 重要性打分 → Score(Si) = Σ(weight_j × feature_j)
├─ Top-K选择 → 保留得分最高的2-3句
└─ 逻辑重组 → 按时间/因果顺序排列
    ↓
压缩结果（80字，信息保留率85%+）
```

### 3.2 冗余去除：消除"信息熵"浪费

**冗余类型识别**：

| 类型 | 特征 | 处理策略 |
|:---|:---|:---|
| **重复陈述** | 同一事实多次出现 | 首次保留，后续删除 |
| **套话空话** | "众所周知"、"一般来说" | 直接删除 |
| **过度细节** | 与决策无关的具体数字 | 保留范围/趋势，删除精确值 |
| **背景铺垫** | 长篇历史回顾 | 压缩为一句话摘要 |
| **假设推演** | 多种情景假设 | 只保留最可能情景 |

**示例：信贷报告压缩**

```
【原始】
"该企业成立于2010年，注册资本1000万元，实缴资本1000万元，注册地址位于XX市XX区XX路XX号，法定代表人张三，统一社会信用代码913XXXXXXXXXXXX。企业主要从事电子产品制造，拥有员工500余人，其中研发人员占比15%，2023年营业收入1.2亿元，净利润800万元，同比增长5%。"

【压缩后】
"电子制造企业（2010年成立，实缴1000万），2023年营收1.2亿/净利800万（+5%），员工500人（研发15%）。"
```

**压缩效果**：120字 → 45字，核心信息保留率95%。

### 3.3 层次摘要：从"平面"到"立体"

**核心思想**：不同决策者需要不同粒度，构建可钻取的层次结构。

```
┌─────────────────────────────────────────┐
│  L1 执行摘要（50字）                      │
│  → 一句话结论：批准/拒绝/审慎+核心依据      │
├─────────────────────────────────────────┤
│  L2 关键指标（200字）                     │
│  → 5-8个核心指标：负债率、逾期次数、担保余额等 │
├─────────────────────────────────────────┤
│  L3 风险信号（500字）                     │
│  → 分类风险：信用/市场/操作/合规，各Top-3   │
├─────────────────────────────────────────┤
│  L4 详细证据（1500字）                    │
│  → 关键句原文+出处，支持追溯验证           │
├─────────────────────────────────────────┤
│  L5 完整原文（链接）                      │
│  → 不进入上下文，保留引用路径              │
└─────────────────────────────────────────┘
```

**动态加载策略**：

| 决策阶段 | 加载层级 | Token消耗 | 用途 |
|:---|:---|:---:|:---|
| 初筛 | L1+L2 | ~300 | 快速过滤明显不合格 |
| 详审 | L1+L2+L3 | ~800 | 全面风险评估 |
| 疑案 | L1-L4全量 | ~2000 | 深度分析+溯源 |
| 存证 | L5链接 | ~50 | 合规留痕，按需展开 |

## 四、金融场景实战

### 4.1 征信报告压缩（20页→核心指标）

**原始结构**：
```
第1-3页：  企业基本信息（注册、股东、变更记录）
第4-8页：  信贷记录（未结清、已结清、对外担保）
第9-12页： 公共记录（欠税、行政处罚、诉讼）
第13-15页： 查询记录（近期被查询情况）
第16-20页： 声明信息（异议标注、特殊说明）
```

**压缩策略**：

| 模块 | 原始篇幅 | 压缩方法 | 压缩后 |
|:---|:---:|:---|:---:|
| 基本信息 | 3页 | 保留：成立时间、注册资本、实控人、行业分类 | 3句 |
| 信贷记录 | 5页 | 提取：未结清笔数/余额、逾期记录、担保余额 | 表格 |
| 公共记录 | 4页 | 提取：欠税金额、处罚次数、诉讼标的 | 列表 |
| 查询记录 | 3页 | 统计：近1/3/6个月查询次数，识别"征信花" | 3个数字 |
| 声明信息 | 5页 | 全文保留（通常关键） | 原文 |

**压缩结果示例**：

```markdown
【企业征信压缩摘要】

┌─ 基本信息 ─┐
电子制造│2015年成立│实缴5000万│实控人李某│无变更

├─ 信贷概况 ─┤
未结清：8笔/余额1.2亿│逾期：0│担保：3笔/余额8000万
│关注类：1笔（2023年，展期）│

├─ 公共记录 ─┤
欠税：0│处罚：1次（环保，2022年，已整改）│
诉讼：2起（原告：供应链纠纷，标的120万；被告：无）

├─ 查询记录 ─┤
近1月：2次│近3月：5次│近6月：12次（⚠️ 频繁）

└─ 风险评级 ─┘
综合：BBB（审慎介入）│主要风险：担保圈+查询频繁
```

**Token对比**：15,000字 → 300字，压缩率98%，决策关键信息保留率90%+。

### 4.2 通话记录时序压缩（电子存证）

**场景**：客户投诉调查，需从1000条通话记录中找到关键证据。

**压缩策略**：

```
原始数据：1000条记录 × 平均200字 = 20万字

第一层：时间窗口过滤
→ 投诉前后7天 → 筛选至150条

第二层：参与方过滤  
→ 涉及投诉客户+客服主管 → 筛选至45条

第三层：内容特征提取
→ 关键词命中（投诉、不满、赔偿、升级）→ 筛选至12条

第四层：关键句提取
→ 每通保留：开场确认+核心诉求+处理结论 → 12条×3句

最终结果：36句，约800字，关键证据链完整
```

**可视化呈现**：

## 六、高级策略：智能压缩决策

### 6.1 查询驱动的动态压缩

```
用户查询："这家企业最近有没有逾期？"
    ↓
压缩策略调整：
├─ 优先级提升：信贷记录模块、近期时间窗口
├─ 优先级降低：历史股东变更、早期财报
├─ 提取粒度：逾期记录精确到"笔数+天数+金额"
└─ 保留原文：逾期相关的完整句子（支持溯源）

vs 查询："这家企业主营业务是什么？"
    ↓
压缩策略调整：
├─ 优先级提升：基本信息、行业分类、经营范围
├─ 提取粒度：业务描述保留完整段落
└─ 压缩重点：财务数据仅保留规模量级
```

### 6.2 增量压缩与缓存

```
首次处理：20页征信报告
    ↓
全量压缩 → 生成L1-L5全层次摘要 → 缓存

3天后更新：新增1条查询记录
    ↓
增量处理：仅压缩新增部分 → 合并到已有摘要 → 更新缓存

优势：
- 处理延迟：从5秒降至0.5秒
- Token成本：降低95%
- 实时性：T+0更新
```

### 6.3 人机协作的"可展开"压缩

```
AI呈现（压缩态）：
"该企业存在2笔对外担保，余额8000万，占净资产40%[展开]"

用户点击[展开]：
├─ 担保明细：
│  ├─ 被担保方A：3000万，2023-06至2024-06，连带责任
│  └─ 被担保方B：5000万，2023-09至2025-09，一般保证
├─ 风险分析：
│  └─ 被担保方A近期涉诉（点击[查看详情]）
└─ 原始条款：[链接至合同扫描件]

设计原则：
- 首屏只呈现"决策必需"的最小信息
- 次级信息一键可达，不超过2次点击
- 原始证据永远可溯源，满足合规
```
