# 多路召回：打破单一索引的限制——向量+图谱+SQL的并行融合

## 一、背景：单一索引的"能力边界"

### 1.1 一个真实的信贷风控困境

某银行信贷审批系统遇到复杂查询：

**审批场景**：评估"华创科技"的贷款申请

**需要核查的信息**：
- 企业基本信息：行业、规模、成立时间
- 关联关系：实控人、股东、对外投资、担保圈
- 经营状况：近3年营收、现金流、纳税记录
- 风险信号：涉诉、失信、行政处罚、舆情

**单一索引的局限**：

| 索引类型 | 能查到什么 | 查不到什么 |
|:---|:---|:---|
| **向量检索** | 企业简介、行业描述语义相似 | 精确的股权比例、担保金额 |
| **知识图谱** | 关联企业、担保链条 | 实时交易流水、最新财报数据 |
| **关系数据库** | 结构化财务指标 | 非结构化舆情、语义关联 |

**实际查询**："华创科技的实控人控制的其他企业，近半年是否有逾期记录，且涉及新能源行业"

- 向量库：能找"新能源"相关企业，但无法关联"实控人"
- 图谱：能查"实控人-控制"关系，但无法过滤"近半年逾期"
- SQL：能查逾期记录，但无法识别"新能源行业"语义

**核心洞察**：复杂决策需要**多维度信息的交叉验证**，单一索引类型无法覆盖。

### 1.2 多路召回的价值矩阵

```
┌─────────────────────────────────────────┐
│  查询维度 vs 索引能力矩阵                  │
├─────────────────────────────────────────┤
│                                         │
│  语义相似性 ──────→ 向量库（Dense/Sparse） │
│     ↓                                   │
│  关系推理 ────────→ 知识图谱（Graph）      │
│     ↓                                   │
│  精确匹配/聚合 ───→ 关系数据库（SQL）      │
│     ↓                                   │
│  实时/外部数据 ───→ API检索（Search/KG）   │
│                                         │
└─────────────────────────────────────────┘
         ↓
    多路并行 → 结果融合 → 交叉验证
```

## 三、关键技术实现

### 3.1 查询解析与路由

```python
class QueryRouter:
    """查询路由决策器"""
    
    def __init__(self):
        self.patterns = {
            'vector': ['行业', '领域', '类似', '相似', '概念'],
            'graph': ['关联', '控制', '股东', '担保', '上下游', '实控人'],
            'sql': ['金额', '大于', '小于', '逾期', '营收', '占比', '近N个月'],
            'api': ['最新', '实时', '新闻', '舆情', '今日']
        }
    
    def route(self, query: str) -> List[str]:
        """决策激活哪些通道"""
        activated = []
        
        for channel, keywords in self.patterns.items():
            if any(kw in query for kw in keywords):
                activated.append(channel)
        
        # 默认至少激活向量通道
        if not activated:
            activated = ['vector']
        
        return activated
    
    def decompose(self, query: str) -> Dict:
        """复杂查询分解为子查询"""
        return {
            'vector_subquery': self._extract_semantic_part(query),
            'graph_subquery': self._extract_relational_part(query),
            'sql_subquery': self._extract_structured_part(query)
        }
```

### 3.3 结果融合策略

| 融合模式 | 适用场景 | 算法 |
|:---|:---|:---|
| **并集融合** | 高召回优先，不漏信息 | 去重后全部保留 |
| **交集融合** | 高精度优先，交叉验证 | 多通道共同命中 |
| **加权融合** | 平衡精准 | 通道权重×相关性 |
| **级联过滤** | 性能优先 | 逐层精化 |

**加权融合示例**：

```python
def weighted_fusion(channel_results: Dict, weights: Dict) -> List[Dict]:
    """多通道结果加权融合"""
    
    entity_scores = defaultdict(lambda: {
        'total_score': 0,
        'channel_hits': [],
        'properties': {}
    })
    
    for channel, results in channel_results.items():
        weight = weights.get(channel, 1.0)
        
        for entity in results:
            entity_id = entity['canonical_id']
            
            # 加权得分
            entity_scores[entity_id]['total_score'] += \
                entity['relevance'] * weight
            
            # 记录来源
            entity_scores[entity_id]['channel_hits'].append(channel)
            
            # 属性合并（图谱优先关系，SQL优先指标）
            entity_scores[entity_id]['properties'].update(
                entity.get('properties', {})
            )
    
    # 多样性boost：多通道命中的实体加分
    for eid, data in entity_scores.items():
        diversity_bonus = len(data['channel_hits']) * 0.1
        data['total_score'] *= (1 + diversity_bonus)
    
    # 排序返回
    sorted_entities = sorted(
        entity_scores.items(),
        key=lambda x: x[1]['total_score'],
        reverse=True
    )
    
    return sorted_entities
```

## 五、架构优化与生产实践

### 5.1 性能优化策略

| 优化点 | 策略 | 效果 |
|:---|:---|:---:|
| **并行化** | 三路召回异步执行，非串行 | 延迟从3×降至1.2×最慢通道 |
| **缓存层** | 热点查询结果缓存（实体级） | 命中率60%，平均延迟-50% |
| **预计算** | 图谱关系快照、聚合指标定时更新 | 复杂查询从秒级降至毫秒级 |
| **降级策略** | 某通道超时，其他通道结果兜底 | 可用性99.9% |

### 5.2 一致性保障

```
【多源数据一致性处理】

场景：图谱显示"华创科技"注册资本5000万，SQL显示3000万

处理流程：
1. 冲突检测：字段级对比，标记差异
2. 置信度评估：
   ├─ 图谱数据来源：工商信息（2023-06抓取）
   └─ SQL数据来源：客户填报（2024-01录入）
3. 时效判断：图谱数据较旧，SQL可能未更新
4. 验证动作：实时调用工商API查询最新数据
5. 结果：最新为5000万（2023-09增资），更新SQL，标记图谱待刷新
6. 呈现：显示"5000万（2023-09增资，原3000万）"，附变更记录链接
```

### 5.3 查询复杂度分级

| 级别 | 特征 | 激活通道 | 处理策略 |
|:---|:---|:---|:---|
| **简单** | 单一条件，明确实体 | 1路（通常为SQL） | 直接查询，<100ms |
| **中等** | 2-3个条件，需关联 | 2路（SQL+图谱或向量） | 并行查询，200-500ms |
| **复杂** | 多条件交叉，语义+结构+关系 | 3路全量 | 并行+融合，500ms-2s |
